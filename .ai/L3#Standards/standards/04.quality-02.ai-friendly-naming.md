# 04.quality-02.ai-friendly-naming - AI 友好命名扩展规范

> **文档版本**：v1.0.0
> **创建日期**：2026-05-11
> **父规范**：`04.quality-00.naming-philosophy.md`、`04.quality-01.naming.md`、`03.frontend-01.structure.md`

---

## 📍 文档定位

本文档把 ZERO 的命名哲学固化为一条可执行的强约束：

**命名长度不是成本。AI 友好 > 人类打字方便。结构清晰完整 > 书写简短。**

凡是在 `04.quality-00.naming-philosophy.md` 中已经倾向但仍可能被解读为"建议"的规则，在这里升级为硬约束。

---

## 🎯 强约束（零容忍）

### 1. 长名字是正确默认

- **允许且鼓励**：40-60 字符的文件名、函数名、变量名。
- **不允许**：为了"短一点"在命名里省掉关键语义段。
- **例**：`UsersList.BatchImportFromCsv.view.vue` 合法；`UsersList.bic.view.vue` 禁止。

### 2. 结构要素不允许省略

文件名必须完整表达：**主体领域（FeatureName）+ 子能力（SubFeature，可选）+ 角色层（FileType/layer）+ 扩展名（ext）**。

- 缺主体：`list.view.vue` 禁止。必须写 `UsersList.view.vue`。
- 缺角色：`UsersList.vue` 禁止。必须写 `UsersList.view.vue`。
- 缺子能力而意义不全：`UsersList.view.vue` 同时承担创建/编辑/导入时必须拆成 `UsersList.Create.view.vue`、`UsersList.Edit.view.vue`、`UsersList.BatchImport.view.vue`。

### 3. 平铺优先，结构靠命名而非目录

- 同一领域内的文件默认在同一目录平铺。
- 不通过新增子目录（`components/`、`services/`、`hooks/`、`containers/`）来替代完整命名。
- 需要物理隔离时（例如编译输出、第三方原样副本、特性开关分组）才允许增设子目录，并在页面清单（`manifest_list`）或对应 L5 文档中登记理由。

### 4. 搜索友好是硬指标

以下三个动作必须在 1 秒内可用文件名前缀/后缀完成：

- 列出领域的全部文件：按 `FeatureName.` 前缀搜索。
- 列出角色的全部文件：按 `.layer.ext` 后缀搜索。
- 跨项目迁移：用全名 grep 可唯一定位。

违反 3 任意一条视为命名不合规。

### 5. 抽象化的模糊词禁用

以下词单独作为文件或主体名使用时禁止：`index`（入口除外）、`main`（入口除外）、`common`、`base`、`util`、`helper`、`handler`、`manager`、`service`、`core`。

它们必须搭配具体语义：

- ❌ `utils.ts`
- ✅ `dateTime.util.ts`
- ✅ `stringFormat.util.ts`
- ❌ `BaseForm.vue`
- ✅ `UsersList.EditForm.view.vue`

### 6. 缩写白名单（行业标准）

仅允许使用下列行业标准缩写：`id`、`url`、`api`、`http`、`https`、`html`、`css`、`json`、`sql`、`ip`、`db`、`ui`、`os`、`cdn`、`dns`、`tls`、`jwt`、`oauth`、`sso`、`mfa`、`2fa`、`otp`。

其余一律写完整单词。`04.quality-00.naming-philosophy.md` 的"通用禁止列表"在本规范下视为硬禁用。

### 7. 下划线 vs 连字符 vs 点分

- 目录：全小写 + 连字符：`pages/admin/users-list/`。
- 前端文件（Vue/React/HTML/JS/TS/CSS）：点分命名 `FeatureName.layer.ext`。
- 后端类文件（PHP/Java）：纯大驼峰，不用点分（沿用 `04.quality-01.naming.md`）。
- 数据库表/字段：下划线（沿用 ZERO 原规范）。
- 同一文件名内**禁止**混用 `-` 和 `.`。

## ✅ 正确示例

```
UsersList.view.vue
UsersList.Edit.view.vue
UsersList.BatchImportFromCsv.view.vue
UsersList.logic.ts
UsersList.style.css
UsersListPermissions.const.ts
authentication.api.php
authentication.jwtVerification.service.php
dateTime.util.ts
```

## ❌ 错误示例

```
list.vue            // 缺主体 + 缺 layer
UsersList.vue       // 缺 layer
ul.view.vue         // 缩写主体
users.cfg.ts        // 用缩写 cfg
auth.cmd.php        // 用缩写 cmd
utils.ts            // 抽象名无具体语义
BaseForm.vue        // 抽象名无具体语义
components/UsersList/Edit.vue   // 用目录代替命名
```

## 🔧 前端技术栈指引

### TypeScript / React

- FeatureName 用 PascalCase：`Pricing.DetailSheet.tsx`。
- 非 UI 模块用 camelCase：`pricing.api.ts`、`classic.const.ts`、`app.route.tsx`。
- 允许的 layer：`view`、`shell`、`ui`、`api`、`const`、`route`、`provider`、`hook`、`context`、`type`、`util`、`style`。
- 保留给生态的 layer（禁止自造）：`test`、`spec`、`d`、`module`、`stories`、`config`。
- 入口固定名不加 layer：`main.tsx`、`App.tsx`、`styles.css`、`index.html`、`vite.config.ts`、`tsconfig.json`。

### Vue

- FeatureName 用 PascalCase：`UsersList.view.vue`、`UsersList.Edit.view.vue`。
- layer 保持 `view` / `logic` / `style` / `api`（与 ZERO 既有规范一致）。

### PHP / Java 后端

- 沿用 `04.quality-01.naming.md` 中的纯驼峰：`AuthenticationJwtVerificationService.php`。
- 不使用点分 layer，避免与前端规则冲突。
