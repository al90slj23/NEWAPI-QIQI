# NEWAPI-QIQI 开发规范

> **优先级**：项目级最高优先级  
> **适用范围**：NEWAPI-QIQI / new-api 本地开发、CI 构建、线上部署  
> **创建日期**：2026-07-04

## 0. 核心原则

NEWAPI-QIQI 的代码基线是官方 new-api。所有自定义修改必须尽量小、可回滚、可解释，并且能清楚说明与官方版本的差异。

## 1. Docker 镜像必须默认在 GitHub 构建

**零容忍规则**：禁止默认在业务服务器上构建 Docker 镜像。

业务服务器包括但不限于：

- `qiqi`
- `na.q.srl` 所在服务器
- 承载数据库、Redis、反向代理或生产容器的服务器

允许的默认流程：

1. 本地修改代码。
2. 本地运行必要测试和前端构建校验。
3. 提交到 GitHub。
4. 由 `.github/workflows/ghcr-newapi-qiqi.yml` 构建 Docker 镜像。
5. 镜像推送到 `ghcr.io/lickjcr/newapi-qiqi`。
6. 生产服务器只执行拉取镜像和重启应用容器。

生产服务器允许执行：

- `docker compose pull`
- `docker compose up -d`
- 健康检查
- 日志查看
- 镜像回滚

生产服务器禁止默认执行：

- `docker build`
- `docker buildx build`
- `bun install` 用于生产镜像构建
- `go build` 用于生产镜像构建
- 任何高 CPU/高内存的镜像构建任务

例外规则：

- 只有用户明确批准“就在服务器上构建”时才允许。
- 执行前必须说明风险：CPU、内存、磁盘、网络、服务可用性。
- 执行前必须确认有回滚方案。

## 2. 部署必须先保护线上可用性

线上部署前必须确认：

- 当前容器健康状态。
- 当前镜像标签。
- compose 文件路径。
- 回滚镜像或备份 compose 文件存在。

部署时只重启必要服务。默认只重启 app 容器，不重建数据库、Redis、日志库。

## 3. 修改必须保持官方基线清晰

每次修改必须能回答：

- 基于哪个官方 commit 或官方 release。
- 修改了哪些文件。
- 为什么修改。
- 如何验证。
- 如何回滚。

不允许把历史遗留修改混进新修改。

## 4. 前端构建规则

NEWAPI-QIQI 当前涉及两个前端：

- `web/default`
- `web/classic`

修改 classic 页面时，至少验证 classic 构建；修改共享依赖时，需要验证两个前端。

## 5. 线上故障处理规则

线上不可达时优先恢复服务，而不是继续发布。

顺序：

1. 检查 SSH。
2. 检查 HTTPS。
3. 检查 Docker/compose 状态。
4. 先恢复已有稳定镜像。
5. 再处理新版本部署。

若 SSH 和 HTTPS 同时超时，判断为主机或网络层问题，先走服务器控制台恢复。
