# 09 - Hooks 与规范同步冲突检测

## 核心理念

**规范同步冲突检测是开发方法论**，不依赖特定工具。有 hook 能力的 IDE 自动触发，无 hook 能力的环境手动执行。

---

## Hook 架构原则（零容忍）

`.kiro/hooks/` 是**只读映射目标**，禁止直接修改。

```
❌ 禁止：直接在 .kiro/hooks/ 创建文件
✅ 正确：在 .ai/L0#Execution/hooks/ 创建，再运行 ./gogogo.sh 8a
```

### 目录结构

```
.ai/L0#Execution/hooks/                ← 源文件（唯一可编辑）
  standards-sync-on-complete.json
  pre-commit-typecheck.json
  husky/                     ← Git hook 源文件
  ide/                       ← 其他 IDE hook 源文件

.kiro/hooks/                 ← 映射目标（只读）
  standards-sync-on-complete.kiro.hook  → symlink
  pre-commit-typecheck.kiro.hook        → symlink
```

### 文件格式

**源文件**：`.json`（人类友好）
**映射后**：`.kiro.hook`（自动重命名）

**必须字段**：

```json
{
  "version": "1.0.0",
  "enabled": true,
  "name": "hook名称",
  "description": "hook描述",
  "when": { "type": "agentStop" },
  "then": { "type": "askAgent", "prompt": "..." }
}
```

## 规范同步冲突检测（开发方法论级）

### 设计理念

**核心价值**：通过每次规范变更，触发全局冲突检测，逐步收敛到一致性。

**工作原理**：

```
触发点（规范需要调整）
    ↓
横向扫描：L4 内所有相关文档（找历史矛盾）
    ↓
纵向对比：L4 → L3 → L2 → L1（找层级不一致）
    ↓
列出所有冲突点 → 等待用户确认
    ↓
批量修正（同步所有关联文档）
```

**最终效果**：

- ✅ 所有文档规范一致
- ✅ 没有历史遗留矛盾
- ✅ 新旧规范不会并存
- ✅ AI 读取任何文档都能得到正确指引

### 实施方式

#### 方式 1：自动触发（支持 hook 的 IDE）

**适用**：Kiro、Claude Code、OpenCode

**配置**：`.ai/L0#Execution/hooks/standards-sync-on-complete.json`

#### 方式 2：手动执行（不支持 hook 的环境）

**适用**：Cursor、CLI、其他 IDE

**执行时机**：每次规范变更后手动执行

### 冲突检测流程

#### 第一步：判断是否涉及规范变更

本次任务是否涉及规范、架构、数据库、API、命名规则的新增或变更？
- 否 → 无需执行
- 是 → 继续第二步

#### 第二步：冲突检测（必须执行）

**1. L3#完整规范 (Standards) 内部冲突检测**

搜索所有相关 md 文件，列出矛盾点：

```bash
grep -r "关键词" .ai/L3#Standards/standards/
```

**2. L2#规范索引 (Index) 与 L3#完整规范 (Standards) 一致性检测**

对比索引与完整规范，列出不一致点：

```bash
grep "关键词" ".ai/L2#Index/toc.md"
grep -r "关键词" ".ai/L3#Standards/standards/"
```

**3. L2 索引一致性检测**

检查索引是否覆盖本次主题：

```bash
grep "关键词" ".ai/L2#Index/toc.md"
```

**4. L1 宪法级检测**

判断是否需要写入零容忍规则：

```
符合条件：
- 涉及核心架构原则
- 违反会导致严重后果
- 需要全员强制遵守
```

#### 第三步：执行同步（确认后）

用户确认后，按以下顺序更新：

```
1. 更新 L4 所有冲突文件（统一口径）
2. 同步 L3 对应文件（提取核心要点）
3. 更新 L2 索引（补充缺失主题）
4. 更新 L1 零容忍规则（如需要）
5. 运行 ./gogogo.sh ai（同步到所有 IDE）
```

## 新增 Hook 流程（仅支持 hook 的 IDE）

```bash
# 1. 创建源文件
touch .ai/L0#Execution/hooks/my-new-hook.json

# 2. 编辑配置（见上方格式）

# 3. 映射到 IDE
./gogogo.sh 8a

# 4. 验证
ls -la .kiro/hooks/my-new-hook.kiro.hook
```

## Hook 类型（仅支持 hook 的 IDE）

| 类型 | 触发时机 | 示例 |
|------|---------|------|
| `agentStop` | 任务完成后 | 规范同步检查 |
| `userTriggered` | 手动触发 | 提交前类型检查 |
| `fileEdited` | 文件保存后 | Vue 文件变更后同步数据库 |
| `preToolUse` | 工具使用前 | 写入操作前确认 |

## 强制执行规范

1. **禁止直接修改映射目标**：`.kiro/hooks/` 是只读
2. **规范同步冲突检测不可跳过**：有 hook 启用，无 hook 手动执行
3. **冲突检测必须完整执行**：必须完成四步检测流程
4. **用户确认后才执行同步**：不要自动修复冲突

---

## 核心原则

### 架构原则

- `.ai/L0#Execution/hooks/` 是 Hook 唯一主版本（平台无关源文件）
- `.kiro/hooks/` 是只读映射目标（符号链接）
- 所有 Hook 通过 `./gogogo.sh 8a` 同步
- 源文件用 `.json`，映射后自动变成 `.kiro.hook`

### 规范同步原则（开发方法论级）

- 每次规范变更 = 一次全局纯化
- 横向扫描 L4 内部冲突
- 纵向对比 L4→L3→L2→L1 一致性
- 用户确认后才执行同步
- 逐步收敛到一致性
- **不依赖特定工具**：有 hook 自动触发，无 hook 手动执行

### 冲突检测原则

- 不要自动修复冲突
- 列出所有矛盾点
- 等待用户确认
- 批量修正所有关联文档
