## 核心架构原则（零容忍级）

### 1. 源文件直接放在 L0 子目录

```
✅ 正确：
.ai/L0/hooks/
  ├── standards-sync-on-complete.json
  └── pre-commit-typecheck.json

.ai/L0/skills/
  └── {skill-name}/
      └── SKILL.md

❌ 错误：
.ai/L0/hooks/kiro/
  └── standards-sync-on-complete.json

.ai/L0/skills/kiro/
  └── {skill-name}/
```

### 2. 通过符号链接映射到各 IDE

```bash
./gogogo.sh 8a   # 映射到 Kiro
./gogogo.sh 8f   # 映射到所有工具
./gogogo.sh 8g   # 移除所有映射
```

映射时可能转换格式：
- `.ai/L0/hooks/*.json` → `.kiro/hooks/*.kiro.hook`
- `.ai/L0/skills/{name}/` → `.kiro/skills/{name}/`

### 3. 禁止直接修改映射目标

```
❌ 禁止：直接在 .kiro/hooks/ 或 .kiro/skills/ 创建/修改文件
✅ 正确：在 .ai/L0/ 进行所有修改，符号链接自动生效
```

## 架构设计理念

### 为什么不套平台子目录？

1. **平台无关性**：L0 是平台无关的源文件层，不应包含平台特定的目录结构
2. **简化管理**：所有配置文件直接在 L0 子目录下，一目了然
3. **映射灵活性**：通过符号链接映射，可以灵活支持多个 IDE/工具
4. **格式转换**：映射时可以转换格式（如 `.json` → `.kiro.hook`），无需在源文件层处理

### 为什么用符号链接？

1. **单一数据源**：源文件只有一份，避免多份拷贝导致不一致
2. **自动同步**：修改源文件后，所有映射目标自动生效
3. **只读保护**：映射目标是符号链接，提醒开发者不要直接修改
4. **多工具支持**：同一份源文件可以映射到多个 IDE/工具
